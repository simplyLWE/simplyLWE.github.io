Getting Perspective.

What is this all about?

What is a variable, really? Variables are one of the first concepts we’re introduced to when we’re learning to code. But there is often a lot more to it than meets the eye. And so we are going to take a deeper look at variables to get a better perspective of what is happening when we use them.

Variables

When we declare variables, we usually do something like this: int var = 7;

First we specify the type of data1, then we specify the name we want to call the variable, and finally we assign the value to the variable we want to store.

Memory

So what are we looking at here? Well it turns out this whole thing, int var = 7; is really just a named object that happens to correspond to a block in memory at a particular address. In other words, we can think of variables as a named memory location that stores a value of a specific type.

Pointers

So now we have a value of a specified type stored in a block of memory at a particular address. But now the question is, how do we access this memory location? The answer to that is a pointer. A pointer simply stores a memory address.2 We can think of a pointer as something that “points to” a particular block in memory.

Example

Using this example, int var = 7;

We first have to make a request to the operating system for a block of memory large enough to hold an integer3, and we will initialize this integer to the value 7: new int(7);

Now we can create a pointer to point to that block in memory: int* ptr = new int(7);

But now the question remains, “How do we actually go inside that block of memory to retrieve the value stored there?”

We can do this by “dereferencing” the pointer, like so: *ptr

This is just a fancy, perhaps more concise way for saying, “go get the value stored at that memory address”.

For demonstration purposes, we can simply save the value from the dereferenced pointer into a variable: int var = *ptr;

And notice the two values are the same when we print them to the screen:

So now we can say, int var = 7;

And this is exactly what we started with in the beginning.

Sidenote

I know it seems circular to store the value of a dereferenced pointer into a variable. We did, afterall, start this discussion talking about variables. And even then, pointers themselves are also variables. To be clear, the intent was not to show you how variables are implemented in the C and C++ languages.4 But rather the intent behind this was to build up a picture and to create a better mental model for what is happening inside the machine.

The end.

In the end, when we declare a variable, the compiler allocates space for that variable in memory at a particular address.

A variable is not a value, or something that holds a value, as we humans might think, not really at least. But rather it is an abstraction and a convenient, human-friendly name for a memory location that stores a value of a specific type.

I hope by now you are able to see things from a different perspective. From the machine’s perspective.

P.S. One last note

You might be asking, why does any of this matter? It matters because computers are deterministic machines. And we humans tend to superimpose our bird’s-eye-view on the problems we solve with code. Computers do exactly what we tell them to do. Flaws in our thinking consequently result in buggy code.5 While our code may still compile, run, and fill us with gratification, this is not to say the code is correct or the code produces the intended behavior. Maybe it does provide the intended behavior, but maybe it also introduces unintended behaviors we are not aware of.

So what’s the key takeaway here? Don’t just write code, but write code well. And how do we do that? We think like the machine.

  2. Recall memory addresses are represented in hexadecimal, a number system like any other. Except hex (short for hexadecimal) represents digits using 16 symbols, rather than 10 we humans are so accustomed to. Recall machines use the binary number system. Although binary may be fundamentally simple, a long sequence of bits (ones and zeros) renders it hard to read for the human eye. And so for this reason, binary values are encoded in hex (8 bits is equivalent to 1 byte; 4 bits translates to 1 hex digit; and so 8 bits, or 1 byte, translates to 2 hex digits). And when we are looking at a series of 64-bits in memory, humans can more readily read 16 hex digits easier than 64 binary values, which would otherwise make your eyes cross. ↩︎
